3. 빈번한 통합
- 코드 변경은 프로그램 수명이 끝날 때 많은 수의 코드를 수정하며
발생하는 오버헤드를 피하기 위해 가능한 한 자주 통합해야합니다.
실제로 일부 오픈 소스 프로그램의 경우 매일 밤마다 코드 병합이
자동적으로 이루어지는 야간 빌드를 갖고 있기도 합니다.
4. 여러 버전
- 개발자가 소프트웨어를 만들 떄는 최소 2가지 이상의 버전의
소프트웨어를 만드는 것을 염두에 둬야 합니다. 구체적으로는
더 많은 기능을 갖췄지만 버그가 존재하는 개발 버전과 더 적은 기능이지만
버그가 거의 없는 안정된 버전으로 말이죠. 개발 버전은 아직 완전히
테스트되지 않은 코드를 사용할 위험을 감수하려는 사용자들을 위해서인데
그들은 1번에서 얘기한 것처럼 공동 개발자의 역할을 수행하며
버그 리포트를 수행하고 버그 수정까지 함께 진행합니다.
5. 높은 모듈화
- 소프트웨어의 일반적 구조는 독립적인 구성 요소에 대한 병렬 개발을
허용하는 모듈식 구조를 갖추고 있어야 합니다.
6. 동적 의사 결정 구조
- 공식/비공식적 의사 결정 구조가 필요하고, 이는 변화하는
사용자 요구 사항 및 기타 요소에 따라서 전략적 결정을 내립니다.
하지만 바자 모델 공개 이후 무료/공개 소스 코드를 분석한 결과 코드의 대부분은
전체 사용자의 일부만 써왔기 때문에 전체 사용자의 요구 사항을
반영하는 데에 어려움이 있어 민주적이지 않습니다.
